How to define your own display¶

IHaskell allows you to define a custom display of any of your data types. You simply have to implement the type class IHaskellDisplay, which is defined as

class IHaskellDisplay a where
  display :: a -> IO Display

where Display is composed of pieces of DisplayData:

data Display = Display [DisplayData]
             | ManyDisplay [Display]

data DisplayData = DisplayData MimeType Text

data MimeType = PlainText
              | MimeHtml
              | MimePng Width Height
              | MimeJpg Width Height
              | MimeSvg
              | MimeLatex
              | MimeJavascript

Defining a custom display for OptIn¶

Instead of displaying the data constructor name (such as Yes), let's color the data that it wraps, by generating HTML.

And since we love collecting people's personal data, let's color a Yes as green, and a No as red.

For Ignore, we display a frowny face Unicode character here because we can. We could also insert an image or an animation or some such thing.

In [52]:
import IHaskell.Display

instance IHaskellDisplay OptIn where
  display action = return $ Display [plain $ show action, html $ code action]
    where
      code action = concat ["<span style='font-weight: bold; color:"
                           , colorIt action
                           , "'>"
                           , showString action
                           , "</span>"]
                           
      colorIt (Yes _) = "green"
      colorIt (No _)  = "red"
      colorIt Ignore  = "blue"
      
      showString (Yes y) = show y
      showString (No n)  = show n
      showString Ignore  = ""

Design tradeoffs when writing displays¶

It is not necessarily wise to write a display instance for a data type, because you can have only one instance for the data type. What if you want to sometimes display something one way, and sometimes display it another?

There are two options:

  • Use Haskell's newtype as a hack to define a synonym for a type that can be treated as a separate type so you can define an instance for it.
  • Just explicitly call a display function rather than using display of the IHaskellDisplay type class.

What if we want to display as SVG?¶

Since IHaskell can display SVG, all we need to do is transform the SVG data (represented in a type) into valid XML (represented in a type) into a string that is finally interpreted by IHaskell using the svg function.

In [65]:
asciiTextToSVGString :: Text.Text -> String
asciiTextToSVGString = XML.showTopElement
                       . Svg.xmlOfDocument
                       . AsciiDiagram.svgOfDiagram
                       . AsciiDiagram.parseAsciiDiagram

svg $ asciiTextToSVGString thing
Automatic display another way¶

What if we wanted to display automatically without calling svg and our helper function asciiTextToXMLString?

We can use the newtype trick to define an instance of IHaskellDisplay for a wrapper type AsciiDisplay and then wrap the text when we want it to be displayed as an SVG diagram.

In [66]:
newtype AsciiDisplay = AsciiDisplay Text.Text

instance IHaskellDisplay AsciiDisplay where
  display (AsciiDisplay t) = return $ Display [ plain $ show $ Text.unpack t
                                              , svg $ asciiTextToSVGString t
                                              ]

Doing even better using GraphViz¶

We can do better!

Let's

  • Transform a tree into a graph using the functional graph library.
  • Lay out the graph using the GraphViz library.
  • Generate SVG.

(The code is presented here without much explanation. A future Pittsburgh Haskell session will discuss Foldable, Traversable, and other topics, using detailed visualization, with IHaskell Notebook, of all the intermediate steps.)

In [68]:
import Prelude hiding (foldr)
import Data.Foldable (foldr)
import Data.Traversable (mapAccumL)
import Data.Functor ((<$>))
import System.IO.Strict (hGetContents)

import qualified Data.Graph.Inductive.Graph as G
import qualified Data.Graph.Inductive.PatriciaTree as PatriciaTree

import qualified Data.GraphViz as GV

-- | Label a tree with 1 .. size of tree.
-- Don't care so much about the order, just uniqueness.
labelTree :: T.Tree a -> T.Tree (G.LNode a)
labelTree t =
  let (_, result) = mapAccumL labelNode 1 t
  in result

-- | Create an integer-labeled node, then increment the index.
labelNode :: G.Node -> a -> (G.Node, G.LNode a)
labelNode i a = (i+1, (i, a))

-- | For each labeled node, annotated with its edges to its children.
-- Don't bother labeling the edges.
childEdged :: T.Tree (G.LNode a)
           -> T.Tree (G.LNode a, [G.LEdge ()])
childEdged (T.Node x@(parentNode, _) ts) =
  let edgedSubtrees = childEdged <$> ts
      childEdges = [ (parentNode, childNode, ())
                   | T.Node ((childNode, _), _) _ <- edgedSubtrees
                   ]
  in T.Node (x, childEdges) edgedSubtrees

treeToGraph :: G.Graph gr => T.Tree a -> gr a ()
treeToGraph t =
  let fullyLabeledTree = childEdged $ labelTree t
      (nodes, edges) = foldr combineValues ([], []) fullyLabeledTree

      combineValues (labeledNode, childEdges) (allNodes, allEdges) =
        ((labeledNode : allNodes), childEdges ++ allEdges)
  in G.mkGraph nodes edges

treeToPatriciaGraph :: T.Tree a -> PatriciaTree.Gr a ()
treeToPatriciaGraph = treeToGraph

treeToDotGraph :: GV.Labellable a => T.Tree a -> GV.DotGraph G.Node
treeToDotGraph = GV.graphToDot (GV.nonClusteredParams {
                                   GV.fmtNode = \(_, a) -> [GV.toLabel a]
                                   })
                 . treeToPatriciaGraph

myTreeSVGString <- GV.graphvizWithHandle GV.Dot (treeToDotGraph myTree) GV.Svg hGetContents
svg myTreeSVGString
